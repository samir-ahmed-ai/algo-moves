# Content & Learning Catalog

> Part of the algo-moves world-class roadmap (`docs/roadmap/`). Audience: an autonomous coding agent (often a cheaper model) that will EXECUTE this plan with no prior context. Make it fully self-contained.

## 1. Snapshot — current state

The learning catalog is authored in TypeScript and mirrored into Postgres. The pipeline has three layers, all real and verified:

**Authoring (frontend, source of truth).**
- `frontend/src/content/types.ts` defines the authoring shapes: `CourseDef → TopicDef → ItemDef` (raw) and `Course → Topic → Item` (resolved). An `ItemDef` needs only `id`+`kind`; for `kind:'problem'` items, title/summary/difficulty/source/tags are hydrated from the referenced plugin's `meta` unless overridden (`frontend/src/content/catalog.ts:5-38` `hydrateItem`).
- `frontend/src/content/index.ts` assembles the live `catalog` from `curatedCourses` (`courses.ts`) merged with generated light course defs `IMPORTED_COURSES`, `PREP_COURSES`, `GO_COURSES` from `@/plugins/_generated/courses` (kept out of heavy plugin chunks; regenerated by `npm run build-plugin-meta`).
- `frontend/src/content/catalog.ts:51` `buildCatalog` flattens courses→topics→items and exposes `getItem`, `getTopic`, `breadcrumb`, `adjacent`, `firstItemId`.

**Taxonomy & browse (discovery overlay).**
- `frontend/src/content/taxonomy.ts` hardcodes 5 tracks (`data-structures`, `algorithms`, `design`, `go`, `interview-prep`) and ~30 `CATEGORIES` as inline JS objects. Go categories are the exception — generated from the Go course via `goBrowseCategories` (`taxonomy.ts:216-224`).
- `frontend/src/content/browse.ts` resolves categories→items with dedupe/sort, builds synthetic browse topics, and powers search + breadcrumbs. Tests: `browse.test.ts`, `taxonomy.test.ts`.

**Narrative "Archipelago" initiative.**
- `frontend/src/plugins/imported/story/archipelago.ts` layers a narrative world over the Graphs course: 11 hand-authored `ARCHIPELAGO_REGIONS` + per-problem `PROBLEM_STORY` (Region 1 hand-authored at `archipelago.ts:353-378`; Regions 2–11 from `archipelagoData.generated.ts`) + verified Python/Java `PROBLEM_PORTS`. Only the `graphs` course (`C = 'graphs'`) has a story today.

**Persistence (Postgres mirror).**
- `backend/internal/arcade/migrations/004_content_schema.sql` defines idempotent, additive DDL (no down path): `courses`, `story_regions`, `topics`, `problems`, `items`, `solutions` (per problem×language×file, unique constraint), `tags`+`problem_tags`, `quiz_questions`+`quiz_choices`. Relational quiz tables were chosen deliberately so the "exactly one correct" invariant is queryable (`004:120-121`).
- `frontend/scripts/export-content-sql.mts` is the deterministic seed exporter: it walks all four plugin groups (first-registration-wins dedupe, `export-content-sql.mts:70-113`), emits a single `begin; truncate … restart identity cascade; insert …; commit;` transaction to `db/content_seed.sql` (currently ~2.5 MB), and supports `--check` to fail CI when stale.
- `backend/internal/arcade/content.go` serves public, read-only GET endpoints: `/api/content/catalog` (course/topic/item spine, metadata only) and `/api/content/problems/{id}` (full problem + tags + solutions + quiz), registered at `backend/internal/arcade/api.go:76-77`.

**Guardrails already wired into `npm run check:all`** (`frontend/package.json`): `check-content-sql` (seed drift), `check-plugin-meta`, `check-simulators`, `check-prep-sim-coverage`, `check:quiz-labels`. Separately `npm test` runs `check-orphans` (`frontend/scripts/check-orphans.mjs`).

**Maturity: Functional.** The pipeline is clean, typed, and drift-guarded at commit time, but it has no runtime catalog-integrity validation (unbound plugin ids, prereq cycles pass silently), no deploy-time seed application (`SEED_CONTENT` is manual — no `RUN_CONTENT_SEED` in the backend), narrative content covers exactly one course, and taxonomy is hardcoded rather than data-driven.

## 2. Strengths to preserve

- **TS is the single source of truth; SQL is a derived artifact.** The exporter is deterministic and idempotent (`export-content-sql.mts:13-14`, truncate+reload transaction). Never hand-edit `db/content_seed.sql`. Keep this direction — do not let SQL and TS diverge into two sources.
- **Hydration from plugin meta** (`catalog.ts:hydrateItem`) — authors write minimal `ItemDef`s; metadata flows from `getPluginMeta`. Explicit fields win. Preserve this "thin manifest" ergonomics.
- **Relational quiz schema** over JSONB (`004:120-138`) — enables queryable label/correctness invariants. Keep.
- **Commit-time drift guard** (`check-content-sql` in `check:all`) prevents the classic "forgot to re-export" bug. Keep and extend, don't replace.
- **Generated light course defs** (`_generated/courses`) keep catalog assembly out of heavy plugin chunks — good chunking hygiene. Preserve the generate step.
- **Public read-only content API** mirrors the leaderboard/canvas read pattern (`content.go:12-15`); no auth surface added. Keep read-only.

## 4. The world-class bar

World-class content tooling for a teaching platform (think **freeCodeCamp's curriculum JSON + validation**, **Exercism's `exercises/` manifests with CI-verified metadata**, and **LeetCode's tag/difficulty faceting**) looks like:

1. **Authoring stays in one place, machine-verified.** A single `checkCatalogIntegrity(catalog, pluginMeta)` runs in `check:all` and proves: every `kind:'problem'` item binds to a real plugin id; every `prereqs` edge references an existing item; the prereq graph is a DAG (cycle → CI fail); every browse category resolves to ≥1 problem. Nothing ships that the sidebar can't render.
2. **Derived artifacts are checksum-locked to their schema.** The exporter reads the column list from a shared shape (or embeds a schema hash of `004`) so a schema change that the exporter doesn't know about fails at CI, not at prod `psql`.
3. **Deploy is one-shot and idempotent.** `RUN_CONTENT_SEED=true` on the backend applies `db/content_seed.sql` in the same startup path as `RUN_MIGRATIONS`, inside a transaction, logged. Prod content is never stale relative to a deploy.
4. **Taxonomy is data, not code.** Tracks/categories live in a data file (JSON/TS-const generated the way `GO_BROWSE_CATEGORIES` already is) and are exposed via `/api/content/browse`, so the same taxonomy drives desktop, mobile, and any future client.
5. **Narrative is a general format.** A course→story→regions→problems overlay schema (already half-present as `story_regions`) that any course can opt into by supplying data, not a bespoke `.ts` per world.
6. **Content quality is measured, not just present.** Prep simulators are asserted to emit ≥N frames on their sample input; quiz questions validated for exactly-one-correct and label format (already partially via `check:quiz-labels`).

## 6. Execution-ready backlog

Pick ONE task at a time. Each is self-contained.

1. **Add catalog integrity module.**
   - Goal: Detect unbound plugin ids, dangling prereqs, and prereq cycles.
   - Files: create `frontend/src/content/integrity.ts` exporting `checkCatalogIntegrity(catalog: Catalog, hasPlugin: (id:string)=>boolean): string[]`.
   - Approach: iterate `catalog.items`; for `kind==='problem'` items with a `pluginId`, push an error if `!hasPlugin(pluginId)`. Build an item-id set; for each item, push an error for any `prereqs` entry not in the set. Run DFS colouring (white/grey/black) over the prereq edges to detect a back edge → cycle error. Return `[]` on success.
   - Acceptance test: create `frontend/src/content/integrity.test.ts` — a hand-built catalog with `a.prereqs=['b']`, `b.prereqs=['a']` yields a cycle error; the real `catalog` from `@/content` with `getPluginMeta`-backed `hasPlugin` yields `[]`.
   - Verify: `cd frontend && node_modules/.bin/vitest run src/content/integrity.test.ts`

2. **Wire integrity into check:all.**
   - Goal: Fail CI on catalog integrity errors.
   - Files: create `frontend/scripts/check-catalog-integrity.mjs` (or reuse a vitest file); edit `frontend/package.json` `check:all` to append `&& npm run check-catalog-integrity`.
   - Approach: since integrity needs the built catalog + plugin registry, prefer a vitest test that imports `@/content` and `getPluginMeta` and asserts `checkCatalogIntegrity(...) === []`. Add script `"check-catalog-integrity": "vitest run src/content/integrity.live.test.ts"`.
   - Acceptance test: the live test passes on the current catalog.
   - Verify: `cd frontend && npm run check-catalog-integrity`

3. **Schema-shape checksum in exporter.**
   - Goal: Fail `check-content-sql` when `004`'s column lists drift from the exporter's.
